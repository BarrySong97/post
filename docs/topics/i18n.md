# Renderer i18n

## Responsibility

Desktop renderer UI chrome (menus, buttons, toasts, settings, empty states) is localized with `i18next` + `react-i18next`. Vault/user content (file names, tag names, note bodies) is never translated.

## File Map

- `apps/desktop/src/renderer/src/i18n/index.ts` — singleton init, `resolveLocale`, `changeAppLanguage`, `getStoredLocalePreference`.
- `apps/desktop/src/renderer/src/i18n/locales/zh-CN.json` — default / Chinese strings.
- `apps/desktop/src/renderer/src/i18n/locales/en.json` — English strings (same key tree).
- Settings language row: `apps/desktop/src/renderer/src/pages/settings/settings-page.tsx`.

## Behavior

- Supported preferences: `auto` | `zh-CN` | `en`, persisted under `localStorage` key `post.locale`.
- `auto` follows `navigator.language` (any `zh*` → `zh-CN`, else `en`).
- Fallback language is `zh-CN`.
- Import `@/i18n` from `main.tsx` before first paint so `useTranslation` works on mount.
- Non-React helpers (for example `asset-model.ts`) call `i18n.t(...)` on the singleton.
- Form Zod messages use `createTagFormSchema(t)` / `createViewFormSchema(t)` so error text tracks the active locale when the modal mounts.

## Gotchas

- Call `changeAppLanguage(preference)` from Settings so storage and `i18n.changeLanguage` stay in sync; do not only call `i18n.changeLanguage`.
- Keep zh-CN and en key trees identical; missing keys fall back to zh-CN.
- Legacy inbox label `"待整理"` may still appear in stored asset data; `asset-model` treats it as untagged alongside the translated inbox label.
